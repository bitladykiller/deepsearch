# 05 - 工作流节点详解

## 1. 概述

工作流包含 9 个节点，每个节点都有明确的职责、输入、输出和异常处理策略。本章逐节点详解其完整执行流程。

### 1.1 节点总览

| 节点名 | Agent | 输入 state key | 输出 state key | 是否调用外部服务 |
|--------|-------|---------------|---------------|-----------------|
| `intent` | intent_router | query | intent, draft | 否（纯 LLM） |
| `direct_answer` | direct_responder | query | final, draft | 否（纯 LLM） |
| `plan` | planner | query | plan, outline, sub_questions, search_plan, budget | 否（纯 LLM） |
| `web_search` | scout_web | query, sub_questions, search_plan, supplementary_queries | web_search, web_evidence | 是（Bocha API） |
| `local_rag` | scout_local | query, sub_questions, search_plan, supplementary_queries | local_rag, local_evidence | 是（Milvus） |
| `deep_dive` | evidence_judge | web_evidence, local_evidence, sub_questions | deep_dive, evidence_pool, audit_flags, source_index | 否（纯 LLM） |
| `analyze` | analyst | evidence_pool, audit_flags, sub_questions | analysis, findings, claim_map, needs_more_research, missing_gaps | 否（纯 LLM） |
| `reflect` | planner | query, sub_questions, missing_gaps, search_plan, supplementary_queries | iteration(+1), supplementary_queries | 否（纯 LLM） |
| `write` | writer | query, sub_questions, findings, source_index, audit_flags | draft, final | 否（纯 LLM） |

### 1.2 共享辅助函数（base.py）

所有节点共享 `base.py` 中的辅助函数，按职责分为以下几类：

#### JSON / LLM 交互
| 函数 | 作用 |
|------|------|
| `_last_content(result)` | 从 Agent 结果中提取最后一条消息的文本内容 |
| `_extract_json_block(text)` | 从 LLM 输出中提取 JSON 块（兼容 markdown 代码块） |
| `_load_json(text, fallback)` | 安全解析 JSON，失败返回 fallback |
| `_invoke_json_agent(...)` | 调用 Agent 并解析 JSON 输出，返回 (payload, content, messages) |

#### 规划辅助
| 函数 | 作用 |
|------|------|
| `_default_plan(state)` | 生成默认规划 JSON（兜底） |
| `_guess_primary_entity(query)` | 从查询中猜测主要实体名称 |
| `_derive_direct_search_queries(query)` | 生成直接检索词列表 |
| `_derive_search_plan(...)` | 根据大纲和子问题推导搜索计划 |

#### 查询辅助
| 函数 | 作用 |
|------|------|
| `_extract_query_terms(query)` | 提取关键词（中文 2+ 字符、英文 3+ 字符） |
| `_estimate_relevance(query, text)` | 估算文本与查询的相关性得分（0~1） |
| `_is_query_grounded(candidate, user_query)` | 检查候选查询是否基于用户原始问题 |
| `_build_queries(state, source_preference)` | 从 state 中构建指定来源类型的查询列表 |

#### 记录过滤与处理
| 函数 | 作用 |
|------|------|
| `_is_bad_web_domain(domain)` | 判断是否为低质量网页域名 |
| `_is_official_domain(domain)` | 判断是否为官方或权威机构域名 |
| `_filter_web_records(query, records)` | 过滤网页记录 |
| `_filter_local_records(query, records)` | 过滤本地知识库记录 |
| `_format_raw_records(records, source_type)` | 将原始记录格式化为 JSON 行（LLM 上下文注入） |
| `_minimal_record_filter(records, required_any)` | 最低限度过滤 |
| `_assign_source_ids(records, prefix)` | 为记录列表分配 source_id |
| `_dedupe_sources(items, key_fields)` | 根据指定字段去重 |
| `_enrich_evidence_from_raw(evidence, raw_records)` | 从原始记录补充丢失的字段 |
| `_prune_evidence_to_allowed_sources(evidence, allowed)` | 只保留允许来源 ID 的证据 |
| `_summarize_records(records)` | 生成记录摘要（用于 trace） |
| `_normalize_source_ids(values)` | 归一化来源 ID 列表 |
| `_finalize_query_traces(...)` | 为查询 trace 记录标记 kept/rejected |

#### 证据评分与审计
| 函数 | 作用 |
|------|------|
| `_score_evidence(record)` | 根据来源类型和域名对证据评分 |
| `_fallback_audit(state)` | 证据审计兜底 |
| `_fallback_analysis(state)` | 分析节点兜底 |

#### 引用与报告
| 函数 | 作用 |
|------|------|
| `_extract_citation_ids(content)` | 从正文中提取所有引用 ID |
| `_validate_and_fix_citations(content, valid_ids)` | 校验并修正非法引用 |
| `_render_reference_list(state)` | 渲染参考文献列表 |
| `_render_execution_appendix(state)` | 渲染规划与检索明细附录 |
| `_ensure_reference_section(content, state)` | 确保报告末尾包含参考资料 |
| `_render_fallback_report(state)` | LLM 写作失败时生成兜底报告 |
| `_build_source_lookup(state)` | 构建统一的 source_id → 来源信息 lookup |

## 2. intent 节点（意图识别）

### 2.1 执行流程

```
intent_node(state, agent, agent_name)
    │
    ├── 1. 规则引擎初判
    │   └── detect_intent(state["query"]) → "direct" 或 "multiagent"
    │
    ├── 2. 构造 Prompt
    │   └── "用户问题：{query}\n规则引擎初判：{rule_route}\n请输出 JSON..."
    │
    ├── 3. 调用 Agent
    │   └── _invoke_json_agent(state, prompt, agent, agent_name, "intent", fallback)
    │       ├── agent.invoke({"messages": [human]})
    │       ├── _last_content(result) → raw_content
    │       └── _load_json(raw_content, fallback) → payload
    │
    ├── 4. 解析路由结果
    │   ├── payload.get("route", rule_route)
    │   ├── 校验是否在 {"direct", "multiagent"} 中
    │   └── 异常时回退到规则引擎结果
    │
    └── 5. 返回 state 更新
        └── {"intent": route, "draft": content, "messages": messages}
```

### 2.2 detect_intent 规则引擎

```python
def detect_intent(query: str) -> str:
    normalized_query = query.strip()
    
    # 规则 1：强制 multiagent 的关键词
    force_multiagent_keywords = [
        "调查", "调研", "来源", "证据", "检索统计", "来源清单",
        "重大新闻", "热门项目", "趋势", "新闻", "最新", "盘点",
    ]
    
    # 规则 2：年份 + 趋势词的组合
    if re.search(r"20\d{2}年", normalized_query) and any(
        word in normalized_query for word in ["趋势", "新闻", "调研", "调查", "盘点"]
    ):
        return "multiagent"
    
    # 规则 3：强制关键词
    if any(word in query for word in force_multiagent_keywords):
        return "multiagent"
    
    # 规则 4：一般研究类关键词
    keywords = [
        "调研", "研究", "调查", "盘点", "热门", "趋势", "榜单", "分析",
        "方案", "架构", "设计", "对比", "报告", "代码", "实现", "落地",
        "检索", "知识库", "证据", "来源", "溯源", "资料", "手册", "验证",
        "数据", "模型",
    ]
    return "multiagent" if any(word in query for word in keywords) else "direct"
```

### 2.3 双重判断机制

intent 节点采用**规则引擎 + LLM** 双重判断：

1. **规则引擎**（`detect_intent`）：基于关键词的快速初判
2. **LLM**（intent_router Agent）：理解语义后的二次确认
3. **兜底**：LLM 输出异常时回退到规则引擎结果

这种设计的优势：
- 规则引擎保证了基本的路由正确性（不会把"帮我调查XX"误判为 direct）
- LLM 能处理规则引擎无法覆盖的复杂语义（如"帮我看看最近有什么新闻"）
- 即使 LLM 调用失败，规则引擎也能给出合理的默认路由

## 3. direct_answer 节点（直接回答）

### 3.1 执行流程

```
direct_answer_node(state, agent, agent_name)
    │
    ├── 1. 构造 Prompt
    │   └── "用户问题：{query}"
    │
    ├── 2. 注入记忆上下文
    │   └── with_memory_context(state, prompt)
    │       └── 如果有 memory_context，追加到 prompt 末尾
    │
    ├── 3. 调用 Agent（不解析 JSON）
    │   ├── HumanMessage(content=with_memory_context(state, prompt))
    │   └── agent.invoke({"messages": [human]})
    │
    ├── 4. 提取回答
    │   └── _last_content(result).strip()
    │
    └── 5. 返回 state 更新
        ├── "intent": "direct"
        ├── "final": content          ← 直接写入最终结果
        ├── "draft": content
        ├── "analysis_summary": content
        ├── "needs_more_research": False
        └── "messages": [human, ai_message]
```

### 3.2 与其他节点的区别

- **不解析 JSON**：direct_responder 输出自然语言，不需要 JSON 解析
- **直接写 final**：不经过 write 节点，直接将输出写入 `state["final"]`
- **不进入研究流程**：路由到 direct_answer 后直接到 END

## 4. plan 节点（任务规划）

### 4.1 执行流程

```
plan_node(state, agent, agent_name)
    │
    ├── 1. 生成兜底计划
    │   └── _default_plan(state)
    │
    ├── 2. 调用 Agent
    │   └── _invoke_json_agent(
    │       state,
    │       f"用户需求：{state['query']}\n请先做大纲与问题拆解，再输出规划 JSON。",
    │       agent, agent_name, "plan", fallback
    │   )
    │
    ├── 3. 解析输出
    │   ├── outline = payload.get("outline", fallback["outline"])
    │   ├── sub_questions = payload.get("sub_questions", fallback["sub_questions"])
    │   ├── research_questions = payload.get("research_questions", fallback["research_questions"])
    │   ├── budget = payload.get("budget", fallback["budget"])
    │   └── search_plan = _derive_search_plan(outline, sub_questions, ...)
    │
    └── 4. 返回 state 更新
        ├── "phase": "planning completed"
        ├── "plan": plan_summary (objective)
        ├── "outline": outline
        ├── "sub_questions": sub_questions
        ├── "research_questions": research_questions
        ├── "search_plan": search_plan
        ├── "budget": budget
        ├── "iteration": 0          ← 重置迭代计数
        └── "messages": messages
```

### 4.2 _derive_search_plan 逻辑

搜索计划的生成遵循以下规则：

1. **从用户原始问题生成直接检索词**：
   - 原始问题本身
   - 实体 + "是什么"
   - 实体 + "GitHub"
   - 实体 + "官方文档"
   - 实体 + "使用趋势"
   - 实体 + "AI Agent"

2. **从大纲章节提取搜索词**：
   - 遍历 outline 中每个 section 的 search_queries
   - 检查是否与用户原始问题相关（`_is_query_grounded`）
   - 过滤不相关的搜索词

3. **去重和截断**：
   - 按 query 字段去重
   - 最多保留 6 个搜索计划项

## 5. web_search 节点（网络搜索）

### 5.1 执行流程

```
web_search_node(state, agent, agent_name)
    │
    ├── 1. 构建查询列表
    │   └── _build_queries(state, "web")
    │       ├── 如果有 supplementary_queries（补搜），使用它们
    │       ├── 否则使用原始 search_plan
    │       └── 只保留 source_preference 为 "web" 或 "hybrid" 的项
    │
    ├── 2. 遍历查询，调用 Bocha API
    │   └── for query_index, item in enumerate(queries, 1):
    │       ├── bocha_web_search_records(query_text, count=4)
    │       ├── _assign_source_ids(records, f"{prefix}_{query_index}")
    │       ├── 附加 section_id 和 search_query 到每条记录
    │       └── 追加到 query_traces
    │
    ├── 3. 数据清洗
    │   ├── _dedupe_sources(raw_records, ["url", "title"])
    │   └── _minimal_record_filter(raw_records, ["title", "snippet", "url"])
    │
    ├── 4. 更新检索统计
    │   └── web_retrieval_stats: query_count, raw_count
    │
    ├── 5. 调用 LLM 整理证据（如果 raw_records 非空）
    │   ├── 构造 Prompt（原问题 + 子问题 + 原始网页证据 JSON）
    │   ├── _invoke_json_agent → payload
    │   └── 提取 evidence 列表
    │
    ├── 6. 证据后处理
    │   ├── _prune_evidence_to_allowed_sources → 只保留允许的 source_id
    │   ├── _enrich_evidence_from_raw → 补充 url、domain、title
    │   └── 更新 kept_count, dropped_count
    │
    ├── 7. 追踪记录
    │   └── _finalize_query_traces → 标记每个 trace 的 kept/rejected
    │
    └── 8. 返回 state 更新
        ├── "web_search": summary
        ├── "web_evidence": existing + new（累加）
        ├── "web_retrieval_stats": stats
        └── "web_search_trace": traces
```

### 5.2 迭代标识

```python
iteration = state.get("iteration", 0)
prefix = f"WEB{iteration+1}"
```

- 第 1 轮：source_id 为 `WEB1_1-1`, `WEB1_1-2`, ...
- 第 2 轮（补搜）：source_id 为 `WEB2_1-1`, `WEB2_1-2`, ...

这种命名方式使得不同轮次的证据不会混淆。

### 5.3 _build_queries 的迭代逻辑

```python
def _build_queries(state, source_preference):
    iteration = state.get("iteration", 0)
    
    # 如果是补搜迭代，使用 supplementary_queries
    if iteration > 0 and state.get("supplementary_queries"):
        base_plan = state.get("supplementary_queries", [])
    else:
        base_plan = state.get("search_plan", [])
    
    # 过滤来源类型
    queries = []
    for item in base_plan:
        pref = item.get("source_preference", "hybrid")
        if pref in (source_preference, "hybrid"):
            queries.append(item)
    
    return queries[:6]  # 最多 6 个查询
```

### 5.4 Bocha Web Search API

```python
def bocha_web_search_records(query: str, count: int = 8) -> list[dict]:
```

**调用流程**：
1. 从环境变量读取 `BOCHA_API_KEY`
2. 构造请求体：`{"query": query, "summary": True, "freshness": "noLimit", "count": count}`
3. POST 请求到 `https://api.bocha.cn/v1/web-search`
4. 解析响应中的 `data.webPages` 数组
5. 标准化为记录列表

**返回记录结构**：
```json
{
  "source_id": "WEB-1",
  "title": "页面标题",
  "url": "https://example.com/page",
  "snippet": "页面摘要...",
  "domain": "example.com",
  "source_type": "web",
  "published_at": "2024-01-01"
}
```

**错误处理**：
- `BOCHA_API_KEY` 未配置 → 返回空列表
- HTTP 错误 → 记录日志，返回空列表
- URL 错误 → 记录日志，返回空列表
- JSON 解析错误 → 记录日志，返回空列表

## 6. local_rag 节点（本地知识库检索）

### 6.1 执行流程

与 `web_search` 节点几乎相同，区别在于：

| 对比项 | web_search | local_rag |
|--------|-----------|-----------|
| 数据源 | Bocha API | Milvus 向量数据库 |
| 调用函数 | `bocha_web_search_records` | `search_knowledge_base_records` |
| source_id 前缀 | `WEB{iteration+1}_{query_index}-{n}` | `LOC{iteration+1}_{query_index}-{n}` |
| 证据字段 | 有 url、domain | 有 doc_id，无 url |
| 过滤字段 | url、domain | doc_id、snippet |

### 6.2 search_knowledge_base_records 函数

```python
def search_knowledge_base_records(query: str, limit: int = 5) -> list[dict]:
    if _RAG_SYSTEM is None:
        return []
    return _RAG_SYSTEM.search_records(query, k=limit)
```

底层调用 `RAGSystem.search_records`，执行 Milvus 向量相似度搜索。

## 7. deep_dive 节点（证据裁判）

### 7.1 执行流程

```
deep_dive_node(state, agent, agent_name)
    │
    ├── 1. 前置检查
    │   ├── 如果 web_evidence 和 local_evidence 都为空 → 返回空 dict
    │   └── 否则继续
    │
    ├── 2. 生成兜底审计
    │   └── _fallback_audit(state)
    │       ├── 遍历所有证据，调用 _score_evidence 评分
    │       ├── 检查假设覆盖
    │       └── 返回 evidence_pool, audit_flags, source_index
    │
    ├── 3. 调用 Agent
    │   └── _invoke_json_agent(state, prompt, agent, agent_name, "deep_dive", fallback)
    │
    ├── 4. 合并证据池
    │   ├── LLM 返回的 evidence_pool（可能遗漏部分证据）
    │   ├── 只保留 allowed_source_ids 中的证据
    │   ├── 补充 LLM 遗漏的原始证据（用 _score_evidence 自动评分）
    │   └── 去重（按 source_id）
    │
    ├── 5. 构建 source_index
    │   └── 从 evidence_pool 提取 source_id, label, locator, source_type
    │
    └── 6. 返回 state 更新
        ├── "deep_dive": summary
        ├── "audit": summary
        ├── "evidence_pool": evidence_pool
        ├── "audit_flags": audit_flags
        ├── "source_index": source_index
        └── "messages": messages
```

### 7.2 _score_evidence 评分逻辑

```python
def _score_evidence(record: dict) -> tuple[float, str]:
    source_type = record.get("source_type")
    
    # 本地知识库 → 默认高可信
    if source_type == "local":
        return 0.92, "企业内部知识库证据，默认高可信"
    
    domain = str(record.get("domain", "")).lower()
    
    # 官方/权威域名
    if _is_official_domain(domain):
        return 0.88, "官方或权威机构域名"
    
    # 主流媒体
    if any(word in domain for word in ["news", "finance", "reuters", "bloomberg", "people", "xinhuanet"]):
        return 0.72, "主流媒体域名"
    
    # 普通互联网来源
    if domain:
        return 0.58, "普通互联网来源，需要交叉验证"
    
    # 来源信息不完整
    return 0.45, "来源信息不完整"
```

评分等级：

| 分数 | 等级 | 来源类型 |
|------|------|----------|
| 0.92 | 极高 | 企业内部知识库 |
| 0.88 | 高 | 官方/权威域名（.gov, .edu） |
| 0.72 | 中高 | 主流媒体（reuters, bloomberg 等） |
| 0.58 | 中 | 普通互联网来源 |
| 0.45 | 低 | 来源信息不完整 |

## 8. analyze 节点（分析结论）

### 8.1 执行流程

```
analyze_node(state, agent, agent_name)
    │
    ├── 1. 生成兜底分析
    │   └── _fallback_analysis(state)
    │
    ├── 2. 调用 Agent
    │   └── _invoke_json_agent(state, prompt, agent, agent_name, "analyze", fallback)
    │       Prompt 包含：原问题 + 子问题 + 证据池 + 审计标记
    │
    ├── 3. 解析输出
    │   ├── findings = payload.get("findings", fallback["findings"])
    │   ├── claim_map = payload.get("claim_map", fallback["claim_map"])
    │   ├── needs_more_research = payload.get("needs_more_research", False)
    │   ├── missing_gaps = payload.get("missing_gaps", [])
    │   └── analysis_summary = payload.get("analysis_summary", content)
    │
    └── 4. 返回 state 更新
        ├── "analysis": analysis_summary
        ├── "findings": findings
        ├── "claim_map": claim_map
        ├── "needs_more_research": needs_more_research  ← 控制是否继续迭代
        └── "missing_gaps": missing_gaps
```

### 8.2 关键输出字段

`needs_more_research` 和 `missing_gaps` 是控制迭代循环的关键：

- `needs_more_research = True` + `missing_gaps = [...]` → 路由到 `reflect` 节点
- `needs_more_research = False` → 路由到 `write` 节点

## 9. reflect 节点（反思补搜）

### 9.1 执行流程

```
reflect_node(state, agent, agent_name)
    │
    ├── 1. 提取信息缺口
    │   └── missing_gaps = state.get("missing_gaps", [])
    │
    ├── 2. 构造 Prompt
    │   包含：分析师的 missing_gaps + 原问题 + 子问题 + 已执行搜索 + 已执行补搜
    │
    ├── 3. 调用 Agent
    │   └── _invoke_json_agent(state, prompt, agent, agent_name, "reflect", fallback)
    │
    └── 4. 返回 state 更新
        ├── "iteration": state["iteration"] + 1  ← 递增迭代计数
        └── "supplementary_queries": new_queries
```

### 9.2 补搜计划结构

```json
{
```

### 9.3 reflect 节点的下游

reflect 节点有两条出边：
```python
workflow.add_edge("reflect", "web_search")

```

这意味着 reflect 完成后，web_search 和 local_rag 会再次并行执行，使用新的 `supplementary_queries` 作为搜索计划。

## 10. write 节点（报告撰写）

### 10.1 执行流程

```
write_node(state, agent, agent_name)
    │
    ├── 1. 准备合法引用 ID 列表
    │   ├── valid_source_ids = [item["source_id"] for item in state["source_index"]]
    │   └── 最多 80 个
    │
    ├── 2. 构造 Prompt
    │   包含：核心问题 + 子问题 + findings + source_index + 合法引用 ID + audit_flags
    │   要求：正文使用合法引用 ID，禁止使用不存在的编号
    │
    ├── 3. 调用 Agent
    │   ├── HumanMessage(content=with_memory_context(state, prompt))
    │   └── agent.invoke({"messages": [human]})
    │
    ├── 4. 清理输出
    │   ├── 移除可能的 JSON 代码块标记
    │   └── 移除 markdown 代码块标记
    │
    ├── 5. 引用校验
    │   ├── _validate_and_fix_citations(content, valid_source_ids_set)
    │   ├── 移除非法引用（不在 valid_source_ids 中的）
    │   └── 返回修正后的内容和实际使用的合法引用列表
    │
    ├── 6. 拼接参考文献
    │   └── _ensure_reference_section(content, state)
    │       ├── 检查是否已有 "## 参考资料" 章节
    │       └── 如果没有，调用 _render_reference_list 拼接
    │
    └── 7. 返回 state 更新
        ├── "draft": final_content
        ├── "final": final_content
        └── "messages": messages
```

### 10.2 引用校验机制

```python
def _validate_and_fix_citations(content: str, valid_source_ids: set[str]):
    pattern = r'\[([A-Z]+\d+_\d+-\d+)\]'
    
    def replace_citation(match):
        citation_id = match.group(1)
        if citation_id in valid_source_ids:
            return f"[{citation_id}]"  # 保留合法引用
        else:
            return ""  # 移除非法引用
    
    fixed_content = re.sub(pattern, replace_citation, content)
    used_ids = [cid for cid in _extract_citation_ids(fixed_content) if cid in valid_source_ids]
    return fixed_content, used_ids
```

引用 ID 格式：`[WEB1_1-1]`、`[LOC1_1-3]`

- `WEB` 或 `LOC`：来源类型
- 第一个数字：迭代轮次
- `_{query_index}-{record_index}`：查询序号-记录序号

### 10.3 参考文献列表渲染

`_render_reference_list` 函数的渲染逻辑：

1. 从正文 draft 中按出现顺序提取实际引用的 source_id
2. 如果正文无引用，降级到 findings 中的 source_ids
3. 再降级：使用全量 source_index
4. 对 local 来源按 locator 去重（同一文件多个 chunk 只展示一次）
5. 排列顺序：WEB 在前，LOCAL 在后

```markdown
## 参考资料
- [WEB1_1-1] [web]: 页面标题 | https://example.com
- [WEB1_1-2] [web]: 页面标题 | https://example.com
- [LOC1_1-1] [local]: 文件名 | /path/to/file.md
```
