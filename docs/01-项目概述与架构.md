# 01 - 项目概述与架构

## 1. 项目定位

Deep Research 是一个基于 LangGraph 的多智能体（Multi-Agent）深度研究系统。它的核心能力是：

1. 接收用户的自然语言问题
2. 自动判断是否需要深度研究
3. 将问题拆解为多个子问题
4. 并行执行网络搜索（Bocha API）和本地知识库检索（Milvus 向量数据库）
5. 对检索到的证据进行评分、去重、冲突审计
6. 生成结论并评估证据完备性
7. 若证据不足，自动生成补搜计划并重复检索（最多 N 轮）
8. 最终生成一份带引用标注的 Markdown 格式深度研报

### 1.1 与传统 RAG 的区别

传统 RAG（Retrieval-Augmented Generation，检索增强生成）流程是：
```
用户问题 → 检索 → 拼接上下文 → LLM 生成回答
```

Deep Research 的流程是：
```
用户问题 → 意图识别 → 任务规划 → 并行双源检索 → 证据裁判 → 分析结论
                                         ↑                    │
                                         └── 反思补搜 ←──────┘
                                                        │
                                                   撰写报告
```

关键差异：
- **多轮迭代**：证据不足时自动补搜，而非一次性检索
- **双源并行**：网络搜索和本地知识库同时执行，互为补充
- **证据裁判**：独立的证据评分和冲突审计环节
- **引用追溯**：每条结论都绑定来源 source_id，报告末尾自动生成参考文献

### 1.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| LLM | DashScope ChatTongyi（通义千问） | 国内访问稳定，中文效果好，支持 Function Calling |
| Embedding | DashScope text-embedding-v1 | 与 LLM 同一服务商，免去多 Key 管理 |
| 工作流引擎 | LangGraph (StateGraph) | 原生支持条件路由、状态持久化、checkpointer |
| Web 框架 | FastAPI + Uvicorn | 异步原生，自动生成 OpenAPI 文档 |
| 向量数据库 | Milvus (langchain-milvus) | 分布式向量数据库，支持大规模检索 |
| 关系数据库 | MySQL 8.0 (pymysql) | 记忆系统持久化存储 |
| 缓存 | Redis 7 | 短期记忆缓存 + LangGraph checkpointer |
| 稀疏检索 | BM25+ (rank-bm25 + jieba) | 关键词匹配，与向量检索互补 |
| 前端 | Vue 3 + TypeScript + Vite | 现代前端框架，TypeScript 类型安全 |
| 反向代理 | Nginx | SPA 静态文件服务 + API 反向代理 |
| 容器化 | Docker + Docker Compose | 一键部署所有服务 |

## 4. 核心概念

### 4.1 多智能体（Multi-Agent）

系统包含 8 个专业 Agent，每个 Agent 有独立的：
- **System Prompt**：定义角色、职责、输出格式
- **Temperature**：控制输出随机性（0.0 = 确定性，0.4 = 较随机）
- **工具绑定**：当前所有 Agent 不绑定工具，只做信息抽取

### 4.2 状态图（StateGraph）

LangGraph 的 StateGraph 是整个系统的核心。它定义了：
- **状态结构**（ResearchState）：所有节点共享的数据结构
- **节点**（Node）：每个节点接收 state，返回 state 的部分更新
- **边**（Edge）：节点之间的连接关系
- **条件边**（Conditional Edge）：根据 state 动态决定下一步

### 4.3 迭代研究

系统支持最多 N 轮迭代研究：
```
第 1 轮：plan → web_search ∥ local_rag → deep_dive → analyze
第 2 轮：reflect → web_search ∥ local_rag → deep_dive → analyze
第 3 轮：reflect → web_search ∥ local_rag → deep_dive → analyze
...
最终：write → END
```

每轮的 `analyze` 节点会判断证据是否充足：
- 充足 → 跳转到 `write` 节点撰写报告
- 不足 → 跳转到 `reflect` 节点生成补搜计划，然后进入下一轮

### 4.4 记忆系统

记忆系统分为三层：
1. **短期记忆**：当前会话的消息历史，支持 Redis / MySQL / 内存三种后端
2. **长期记忆**：用户画像、事实知识、历史任务，存储在 MySQL + Milvus
3. **向量索引**：所有长期记忆都会写入 Milvus，支持语义搜索

记忆系统的核心能力：
- **会话摘要压缩**：当消息数超过阈值时，自动调用 LLM 生成摘要
- **自动偏好提取**：检测 "记住"、"我喜欢" 等关键词，自动提取为长期记忆
- **个性化 Prompt 注入**：每次研究前，将用户画像 + 历史记忆 + 对话摘要注入到 Agent 的上下文中
