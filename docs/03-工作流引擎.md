# 03 - 工作流引擎（LangGraph StateGraph）

## 1. 概述

工作流引擎是 Deep Research 的核心，基于 LangGraph 的 `StateGraph` 构建。它定义了从用户输入到最终报告的完整执行路径，包含 9 个节点、2 个条件路由和 1 个迭代循环。

### 1.1 什么是 LangGraph

LangGraph 是 LangChain 生态中的状态图框架，核心概念：

| 概念 | 说明 |
|------|------|
| `StateGraph` | 状态图容器，定义节点和边 |
| `State` | 所有节点共享的 TypedDict 状态结构 |
| `Node` | 处理函数，接收 state，返回 state 的部分更新 |
| `Edge` | 节点之间的固定连接 |
| `Conditional Edge` | 根据 state 动态决定下一步的边 |
| `START` / `END` | 特殊的起始/终止伪节点 |
| `Checkpointer` | 状态持久化存储（Redis / 内存） |

### 1.2 为什么选择 LangGraph

1. **原生条件路由**：支持基于 state 动态决定下一步，完美适配意图识别分流
2. **并行执行**：支持一个节点同时连接到多个下游节点（fan-out），实现并行检索
3. **状态持久化**：通过 checkpointer 实现断点续跑和状态回溯
4. **流式输出**：`stream_mode="updates"` 可以实时获取每个节点的输出
5. **迭代循环**：支持节点之间的循环连接，实现多轮迭代研究

## 2. ResearchState 状态定义

### 2.1 完整结构

```python
class ResearchState(TypedDict):
    # ---- 输入 ----
    query: str                          # 用户原始问题
    user_id: str                        # 用户标识
    tenant_id: str                      # 租户标识
    memory_context: str                 # 注入的跨会话记忆上下文
    
    # ---- 消息历史 ----
    messages: Annotated[List[BaseMessage], operator.add]  # 自动累加的消息列表
    
    # ---- 意图识别 ----
    intent: str                         # 路由结果："direct" 或 "multiagent"
    phase: str                          # 当前阶段标记
    
    # ---- 任务规划 ----
    plan: str                           # 规划摘要（objective）
    outline: list[dict]                 # 大纲列表（章节结构）
    sub_questions: list[str]            # 子问题列表
    research_questions: list[str]       # 研究问题列表
    search_plan: list[dict]             # 搜索计划列表
    budget: dict                        # 资源预算
    
    # ---- 检索结果 ----
    web_search: str                     # 网页检索摘要
    local_rag: str                      # 本地知识库检索摘要
    web_evidence: list[dict]            # 网页证据列表
    local_evidence: list[dict]          # 本地知识库证据列表
    
    # ---- 证据裁判 ----
    evidence_pool: list[dict]           # 合并后的证据池
    deep_dive: str                      # 证据裁判摘要
    audit: str                          # 审计摘要
    audit_flags: list[dict]             # 审计标记（低可信度、冲突等）
    
    # ---- 分析结论 ----
    analysis: str                       # 分析摘要
    needs_more_research: bool           # 是否需要补搜
    missing_gaps: list[str]             # 信息缺口列表
    supplementary_queries: list[dict]    # 补搜计划
    findings: list[dict]                # 结论列表（claim_id + claim + source_ids）
    claim_map: list[dict]               # 结论-来源映射
    source_index: list[dict]            # 来源索引（source_id + label + locator）
    
    # ---- 检索统计 ----
    web_retrieval_stats: dict           # 网页检索统计
    local_retrieval_stats: dict         # 本地检索统计
    web_search_trace: list[dict]        # 网页检索追踪记录
    local_rag_trace: list[dict]         # 本地检索追踪记录
    
    # ---- 输出 ----
    code: str                           # 代码骨架（未使用）
    draft: str                          # 草稿内容
    final: str                          # 最终报告
    
    # ---- 迭代控制 ----
    iteration: int                      # 当前迭代轮次
    max_iterations: int                 # 最大迭代轮次
```

### 2.2 Annotated[List[BaseMessage], operator.add] 的含义

```python
messages: Annotated[List[BaseMessage], operator.add]
```

这是 LangGraph 的**状态累加器**模式：
- `operator.add` 告诉 LangGraph：当节点返回 `{"messages": [...]}` 时，将新消息**追加**到现有列表，而不是替换
- 这允许多个节点各自添加消息，最终 messages 包含所有节点的消息历史
- 其他字段（如 `web_evidence`、`findings`）则是**替换**模式

## 3. 工作流图定义

### 3.1 build_app 函数

```python
def build_app(agents, checkpointer):
    # 1. 创建 StateGraph
    workflow = StateGraph(ResearchState)
    
    # 2. 添加 9 个节点
    workflow.add_node("intent", bind_agent(intent_node, agents.intent_router, "intent_router"))
    workflow.add_node("direct_answer", bind_agent(direct_answer_node, agents.direct_responder, "direct_responder"))
    workflow.add_node("plan", bind_agent(plan_node, agents.planner, "planner"))
    workflow.add_node("web_search", bind_agent(web_search_node, agents.scout_web, "scout_web"))
    workflow.add_node("local_rag", bind_agent(local_rag_node, agents.scout_local, "scout_local"))
    workflow.add_node("deep_dive", bind_agent(deep_dive_node, agents.evidence_judge, "evidence_judge"))
    workflow.add_node("analyze", bind_agent(analyze_node, agents.analyst, "analyst"))
    workflow.add_node("reflect", bind_agent(reflect_node, agents.planner, "planner"))
    workflow.add_node("write", bind_agent(write_node, agents.writer, "writer"))
    
    # 3. 添加边
    workflow.add_edge(START, "intent")
    
    # 4. 添加条件边（意图路由）
    workflow.add_conditional_edges(
        "intent",
        route_after_intent,
        {
            "direct_answer": "direct_answer",
            "plan": "plan",
        },
    )
    
    # 5. 固定边
    workflow.add_edge("plan", "web_search")
    workflow.add_edge("plan", "local_rag")
    workflow.add_edge("web_search", "deep_dive")
    workflow.add_edge("local_rag", "deep_dive")
    workflow.add_edge("deep_dive", "analyze")
    
    # 6. 条件边（分析后路由）
    workflow.add_conditional_edges(
        "analyze",
        should_continue_research,
        {
            "reflect": "reflect",
            "write": "write"
        }
    )
    
    # 7. 迭代循环边
    workflow.add_edge("reflect", "web_search")
    workflow.add_edge("reflect", "local_rag")
    
    # 8. 终止边
    workflow.add_edge("direct_answer", END)
    workflow.add_edge("write", END)
    
    # 9. 编译并返回
    return workflow.compile(checkpointer=checkpointer)
```

### 3.3 Fan-out / Fan-in 模式

LangGraph 中一个节点可以有多个下游边，这形成 **fan-out**（扇出）：

```python
# plan 节点扇出到 web_search 和 local_rag
workflow.add_edge("plan", "web_search")
workflow.add_edge("plan", "local_rag")
```

当 `plan` 节点执行完成后，LangGraph 会**并行**执行 `web_search` 和 `local_rag`。两个节点都完成后，才会进入下一个共同的下游节点 `deep_dive`（**fan-in**，扇入）。

这种模式的优势：
- 网络搜索和本地知识库检索互不依赖，可以并行执行
- 总耗时 = max(web_search_time, local_rag_time)，而非两者之和

## 4. 条件路由

### 4.1 意图路由：route_after_intent

```python
def route_after_intent(state: ResearchState) -> str:
    if state.get("intent") == "direct":
        return "direct_answer"
    return "plan"
```

**输入**：`intent` 节点设置的 `state["intent"]`

**逻辑**：
- `"direct"` → 跳转到 `direct_answer`（简单问答直接回答）
- 其他（包括 `"multiagent"`）→ 跳转到 `plan`（进入深度研究流程）

**返回值**：必须是条件边映射中的 key 之一

### 4.2 分析后路由：should_continue_research

```python
def should_continue_research(state: ResearchState) -> str:
    iteration = state.get("iteration", 0)
    max_iter = state.get("max_iterations", 2)

    # 如果达到最大迭代次数，强制写报告
    if iteration >= max_iter:
        return "write"

    # 如果分析师发现信息缺口，需要补搜
    if state.get("needs_more_research", False):
        return "reflect"

    # 证据充足，写报告
    return "write"
```

**三级判断逻辑**：

```
                    ┌─────────────────┐
                    │ iteration >= max│──── 是 ──→ write
                    └────────┬────────┘
                             │ 否
                    ┌────────▼────────┐
                    │needs_more_      │──── 是 ──→ reflect
                    │research = True? │
                    └────────┬────────┘
                             │ 否
                             ▼
                           write
```

1. **迭代上限检查**：防止无限循环，`iteration >= max_iterations` 时强制写报告
2. **信息缺口检查**：`needs_more_research = True` 时进入 `reflect` 节点生成补搜计划
3. **默认路径**：证据充足，直接写报告

### 4.3 迭代循环的实现

```python
# reflect 节点扇出回到 web_search 和 local_rag
workflow.add_edge("reflect", "web_search")
workflow.add_edge("reflect", "local_rag")
```

当 `analyze` 节点返回 `needs_more_research=True` 时：
1. 路由到 `reflect` 节点
2. `reflect` 节点生成补搜计划（写入 `supplementary_queries`）
3. `reflect` 同时扇出到 `web_search` 和 `local_rag`
4. 两个检索节点检测到 `iteration > 0`，使用 `supplementary_queries` 而非原始 `search_plan`
5. 检索完成后进入 `deep_dive` → `analyze`
6. `analyze` 再次判断是否需要继续

`reflect` 节点中会递增 iteration：

```python
def reflect_node(state, agent, agent_name):
    # ... 生成补搜计划 ...
    return {
        "iteration": state.get("iteration", 0) + 1,  # 递增迭代计数
        "supplementary_queries": payload.get("supplementary_queries", ...),
    }
```

## 5. Checkpointer（状态持久化）

### 5.1 什么是 Checkpointer

LangGraph 的 checkpointer 负责在每个节点执行后保存 state 的快照。它的作用：

1. **断点续跑**：如果执行中断，可以从上次保存的 state 恢复
2. **状态回溯**：可以查看任意节点执行前后的 state
3. **人工介入**：可以在某个节点暂停，等待人工审核后继续

### 5.2 build_checkpointer 函数

```python
def build_checkpointer(config: AppConfig):
    backend = config.checkpointer_backend

    # 优先使用 Redis
    if backend in {"redis", "auto"} and config.enable_memory and config.redis_url:
        from langgraph.checkpoint.redis import RedisSaver
        
        candidate_urls = [config.redis_url]
        # 处理 Redis 认证格式差异
        if "redis://root:" in config.redis_url:
            candidate_urls.append(config.redis_url.replace("redis://root:", "redis://:"))
        
        last_exc = None
        for url in candidate_urls:
            try:
                _CHECKPOINTER_CONTEXT = RedisSaver.from_conn_string(url)
                checkpointer = _CHECKPOINTER_CONTEXT.__enter__()
                checkpointer.setup()  # 创建 Redis 索引
                return checkpointer
            except Exception as exc:
                last_exc = exc
        
        # 处理 RediSearch 不可用的情况
        if last_exc and "FT._LIST" in str(last_exc):
            logger.warning("Redis checkpointer 依赖 RediSearch，已降级")
        else:
            logger.warning("Redis checkpointer 初始化失败，降级内存: %s", last_exc)

    # 降级到内存
    return InMemorySaver()
```

### 5.3 两种 Checkpointer 对比

| 特性 | RedisSaver | InMemorySaver |
|------|-----------|---------------|
| 持久化 | 是（Redis 持久化） | 否（进程退出即丢失） |
| 性能 | 高（Redis 内存数据库） | 极高（纯内存） |
| 依赖 | 需要 Redis + RediSearch 模块 | 无 |
| 适用场景 | 生产环境 | 开发测试 |
| 降级策略 | 连接失败自动降级 | - |

### 5.4 Redis Stack 要求

`langgraph-checkpoint-redis` 依赖 RediSearch 模块（Redis Stack 的一部分），它使用 `FT._LIST` 命令创建全文索引。如果 Redis 不是 Redis Stack 版本，会报错：

```
ResponseError: unknown command 'FT._LIST'
```

此时系统会自动降级到 `InMemorySaver`。

### 5.5 全局上下文管理

```python
_CHECKPOINTER_CONTEXT = None

def cleanup_checkpointer():
    """清理全局 checkpointer 上下文。"""
    global _CHECKPOINTER_CONTEXT
    if _CHECKPOINTER_CONTEXT:
        _CHECKPOINTER_CONTEXT.__exit__(None, None, None)
        _CHECKPOINTER_CONTEXT = None
```

RedisSaver 使用上下文管理器模式（`__enter__` / `__exit__`）。`build_checkpointer` 保存了上下文引用，程序退出时需要调用 `cleanup_checkpointer()` 清理。

## 6. 工作流执行模式

### 6.1 同步执行（invoke）

```python
result = app.invoke(
    state,
    {"configurable": {"thread_id": config.thread_id}},
)
```

- 一次性执行完所有节点
- 返回最终的 state
- 适用于 CLI 模式和同步 API

### 6.2 流式执行（stream）

```python
for update in app.stream(state, config, stream_mode="updates"):
    for node_name, node_output in update.items():
        # 每个节点完成后立即收到其输出
        emit({"type": "phase", "node": node_name, "message": ...})
```

- 每个节点执行完成后立即推送其输出
- 适用于 SSE 流式 API
- `stream_mode="updates"` 表示只推送 state 的增量更新

### 6.3 流式执行的事件格式

```json
// intent 节点完成后
{"intent": {"intent": "multiagent", "draft": "...", "messages": [...]}}

// plan 节点完成后
{"plan": {"phase": "planning completed", "plan": "...", "outline": [...], ...}}

// web_search 节点完成后
{"web_search": {"web_search": "...", "web_evidence": [...], ...}}

// ... 依此类推
```

## 7. bind_agent 函数

### 7.1 作用

```python
def bind_agent(node_func, agent, agent_name: str):
    """将 agent 和 agent_name 绑定到节点函数，返回单参数 callable。"""
    return partial(node_func, agent=agent, agent_name=agent_name)
```

LangGraph 的节点函数签名必须是 `(state: ResearchState) -> dict`，但实际的节点实现需要额外的 `agent` 和 `agent_name` 参数。`bind_agent` 使用 `functools.partial` 将这两个参数预绑定，生成符合 LangGraph 要求的单参数 callable。

### 7.2 调用链

```
LangGraph 引擎调用节点
    → 调用 bind_agent 返回的 partial(state)
        → 实际调用 intent_node(state, agent=..., agent_name=...)
            → _invoke_json_agent(state, prompt, agent, agent_name, ...)
                → agent.invoke({"messages": [human]})
                → 解析 JSON 输出
                → 返回 (payload, content, messages)
```

## 8. 并行执行机制

### 8.1 LangGraph 并行原理

当一个节点有多个下游边时，LangGraph 会并行执行所有下游节点：

```
plan → web_search (并行)
plan → local_rag  (并行)
```

执行引擎会：
1. 将 plan 的输出 state 克隆两份
2. 分别传给 web_search 和 local_rag
3. 等待两个节点都完成
4. 合并两个节点的 state 更新（使用 Annotated 的 operator.add 累加 messages）
5. 传给下一个共同的下游节点（deep_dive）

### 8.2 state 合并规则

- `messages`：累加（`operator.add`），两个节点的消息都会保留
- 其他字段：后执行的节点的值会覆盖先执行的（如果两个节点返回了同一个 key）
- 实际上 `web_search` 和 `local_rag` 写入的 key 不同（`web_evidence` vs `local_evidence`），所以不会冲突

### 8.3 完整的并行执行时间线

```
时间轴 →

plan:     [====]
              │
              ├──→ web_search: [========]
              │                         │
              ├──→ local_rag:  [====]   │
              │                    │    │
              │                    ▼    ▼
              │             deep_dive: [====]
              │                            │
              │                       analyze: [====]
              │
              ▼
         总耗时 = plan + max(web_search, local_rag) + deep_dive + analyze
```

如果没有并行，总耗时将是：
```
plan + web_search + local_rag + deep_dive + analyze
```

并行节省了 `min(web_search, local_rag)` 的时间。
