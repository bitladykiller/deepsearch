```

        ▼
```

## 3. LangGraph 并行与 fan-in 细节

### 3.1 plan/reflect 的 fan-out

```
          plan (或 reflect)
               │
       ┌───────┴───────┐
       ▼               ▼
  web_search        local_rag
  (独立 state 更新)  (独立 state 更新)
       │               │
       └───────┬───────┘
               ▼
           deep_dive
```

LangGraph 语义：
1. 上游节点完成后，两条边并行调度。
2. 两分支各自返回 partial update。
3. 合并规则：
   - `messages`: `operator.add` 累加
   - 其他字段：按 key 合并；同 key 后写覆盖（web/local 写不同 key，通常不冲突）
4. 两分支都结束后才进入 `deep_dive`。

### 3.2 deep_dive 的“空返回等待”逻辑

```python
if not state.get("web_evidence") and not state.get("local_evidence"):
    return {}
```

含义：
- 若 fan-in 时两边都还没有任何证据（或都被过滤空），节点直接返回空 dict。
- 空 dict 不改 state，后续 `analyze` 会基于空 evidence_pool 走 fallback/低置信结论。
- 在正常并行调度下，deep_dive 只在两分支完成后触发一次；该判断主要是防御性保护，而非轮询等待。

### 3.3 迭代时证据“追加不覆盖”

```
第1轮 web_evidence = [WEB1_*]
第2轮 web_search 返回 existing + new
     web_evidence = [WEB1_*, WEB2_*]

deep_dive 每轮重建 evidence_pool（基于当前全部 web+local）
source_index 同步重建
findings 由 analyze 重新生成（不跨轮累加 findings 列表，而是重算）
```

### 3.4 条件路由的精确判定顺序

```
should_continue_research(state):
  1) if iteration >= max_iterations → write   # 硬上限优先
  2) if needs_more_research → reflect
  3) else → write
```

含义：
- 即使分析师仍觉得不够，到上限也强制写报告，防止死循环。
- `iteration` 在 `reflect` 中 +1，不在 `analyze` 中增加。
- `max_iterations` 默认来自配置；API 可覆盖，Schema 限制 1..6。

## 4. 节点 Token 隔离设计（极易误解）

### 4.1 `_invoke_json_agent` 的消息策略

```
 intentionally:
   human = HumanMessage(with_memory_context(state, prompt))
   result = agent.invoke({"messages": [human]})   # 只传当前这一条

 不传:
   state["messages"] 历史

 原因（代码注释）:
   避免 token 随节点累积爆炸
```

### 4.2 那 messages 字段还有什么用？

```
每个节点仍可能 return {"messages": [human, ai]}
ResearchState.messages 使用 Annotated[..., operator.add]
→ 图状态里会累加“审计轨迹”
→ 但下一节点默认不把整段历史塞回 LLM

例外:
  direct_answer / write 同样主要用当前 prompt
  memory_context 才是跨会话个性化通道
```

### 4.3 memory_context 注入位置

```
API/CLI 在 invoke 前:
  memory_context = MemoryManager.build_personalized_prompt_context(...)
  create_initial_state(..., memory_context=...)

每个节点调用 LLM 前:
  with_memory_context(state, prompt)
    → prompt + "\n\n[跨会话记忆]\n" + memory_context

因此:
  记忆是“每节点可见的前缀上下文”
  不是“对话历史逐轮回放”
```

## 6. 记忆系统遗漏算法细节

### 6.1 `extract_memory_from_messages` 真实规则

```
输入: HumanMessage 列表
  │
  ├─ 按 [。！？!?\n；;] 分句
  │
  ├─ facts 关键词命中:
  │    是/有/位于/成立于/负责/使用/叫/
  │    my name is / i am / i'm
  │    → 含关键词且长度>3 的句子加入 facts
  │
  ├─ preferences 关键词命中:
  │    喜欢/偏好/想要/希望/不喜欢/讨厌/
  │    prefer / i like / style
  │    → 同理加入 preferences
  │
  └─ set() 去重后返回 {facts, preferences, tasks:[]}
```

再经 `persist_turn` 二次过滤：
- 问句信号（?？什么吗 how what...）剔除
- 非允许情况下以“你/请问”开头剔除
- 若触发 remember 标记但抽取为空，可把整句 query 当 preference

### 6.2 `format_memories_for_prompt`

```
semantic ≤5 条 → "## 相关背景知识"
episodic ≤3 条 → "## 相关历史任务"
总长度 > max_length → 截断 + 省略提示
```

### 6.3 `calculate_memory_relevance`

```
score =
  0.6 * text_score
+ 0.3 * exp(-age_hours / 168)      # 7 天半衰期
+ 0.1 * min(access_count*0.05, 0.2)

text_score:
  子串包含 → 0.5
  词重叠比例 * 0.8 → 取 max
最终 clamp 到 [0,1]
```

### 6.4 `compress_memories`

```
if len <= target: return
去重: pairwise simple Jaccard > 0.8 视为重复
排序: access_count*10 - age_days
取前 target_count
```

### 6.5 `should_inject_long_term`（存在但主路径未强制）

```
判断短期消息是否为空:
  mysql: SELECT short_term_messages count==0
  redis: LLEN key == 0
  memory: buffer 空

语义意图:
  仅在“新会话第一条”时考虑注入更多长期记忆
  （当前主路径 build_personalized_prompt_context 每次都会取相关记忆，
   该函数更像辅助策略/扩展点）
```

### 6.6 记忆后端选择决策树

```
short_term_backend:
  redis + client ok → Redis list + summary key + TTL + 压缩
  mysql + conn ok  → short_term_messages + summaries 表 + 压缩
  else             → 进程内 ShortTermMemory

long_term_backend:
  disabled → 所有长期写读 no-op/空
  mysql + conn ok → memory_entries + user_profiles +（可选）Milvus 索引
  else → SQLite Semantic/EpisodicStore（开发兜底）

检索优先级（semantic/episodic）:
  Milvus 命中 → 返回
  else MySQL LIKE → 返回
  else SQLite search
```

### 6.7 多租户键空间图

```
Redis:
  ma:short:{tenant}:{user}:{thread}
  ma:short:summary:{tenant}:{user}:{thread}

MySQL:
  WHERE tenant_id=? AND user_id=? AND thread_id=?
  user_profiles PK (tenant_id, user_id)

Milvus memory metadata filter:
  tenant_id / user_id / memory_type / namespace / (optional thread_id)

隔离保证:
  不同 tenant 即使用户名相同也不会串数据
```

## 7. API Schema 与运行时覆盖

### 7.1 ResearchRequest 校验

```
query: str, min_length=1                 # 必填
user_id: default "default_user"
thread_id: default "default_thread"
tenant_id: default "default_tenant"
max_iterations: Optional[int], 1..6      # None → 用服务端 config
enable_memory: Optional[bool]            # None → 用服务端 config
```

### 7.2 覆盖优先级（单次请求）

```
请求体字段（非 null）
  > WorkflowService._base_config（来自文件+环境）
  > 代码默认值

注意:
  前端不传 max_iterations/enable_memory 时，完全吃服务端配置
  CLI --once-query 可用命令行覆盖更多后端开关
```

### 7.3 同步 vs 流式差异细节

```
/run:
  asyncio.to_thread(_run_sync)
  一次性 invoke
  返回 ResearchResponse.final

/stream:
  先 yield status
  后台 Thread 跑 _run_sync_with_events
  app.stream(stream_mode="updates")
  每节点 emit phase
  结束后 emit route + final
  异常 emit error
  最后 __done__ 结束异步生成器

若 stream 过程中 final 仍空:
  代码会再 invoke 一次兜底取 final
  （防 stream 丢字段）
```

## 8. 并发、线程与背压

### 8.1 API 进程内模型

```
Uvicorn 主事件循环（async）
  │
  ├─ /run : to_thread 跑同步图
  │
  └─ /stream :
        主协程 await queue.get()
        worker 线程:
          LangGraph.stream / LLM / HTTP / DB
          emit → run_coroutine_threadsafe(queue.put)

风险点:
  · 重 CPU/阻塞在 worker，不堵 event loop
  · 多并发研究会多线程并行打 LLM/Bocha/Milvus
  · 无全局信号量限流（当前）→ 高并发可能打满外部配额
  · WorkflowService 单例初始化有锁，运行期图实例共享
```

### 8.2 Checkpointer 与 thread_id

```
invoke/stream config = {"configurable": {"thread_id": ...}}

同 thread_id:
  Redis/Memory checkpointer 可能保存检查点轨迹
不同请求若共用 thread_id:
  状态命名空间相同（有意用于连续会话）
新业务会话应换 thread_id
```

### 9.1 后端镜像分层意图

```
1) apt: gcc + default-libmysqlclient-dev + pkg-config

环境:
  PYTHONDONTWRITEBYTECODE=1
  PYTHONUNBUFFERED=1   # 日志立即刷出，便于 docker logs
```

### 9.2 卷与权限

```
app_data → /app/data
  · 容器内 appuser 可写
  · 用于 SQLite/本地数据兜底

config.json / .env 以只读挂载
  · 改配置通常需重启 backend 容器
  · 注意：WorkflowService 懒加载后配置缓存在内存，
    仅重启进程才会重新 from_file
```

### 9.3 Nginx gzip 与 SSE

```
gzip on 对 text/js/css/json 生效
SSE 是 text/event-stream:
  · proxy_buffering off 避免事件被缓冲成一团
  · 长耗时靠 proxy_read_timeout 300s
  · 若研究超过 300s，前端会中断（需调大超时）
```

## 11. 端到端失败矩阵（运维视角）

| 故障点 | 用户可见表现 | 系统行为 | 建议排查 |
|--------|--------------|----------|----------|
| 无 DASHSCOPE_API_KEY | 首次请求 500 | AppConfig.from_file 抛错 | .env / 环境变量 |
| 无 BOCHA_API_KEY | 研报缺网页来源 | web 返回 []，可继续 local | BOCHA_API_KEY |
| MySQL 未就绪 | 记忆降级/告警 | manager 降级 SQLite/内存 | compose ps / logs mysql |
| Redis 非 Stack | checkpoint 降级内存 | 日志提示 FT._LIST | 换 Redis Stack 或接受内存 |
| Milvus 挂 | 本地检索空 | search_records [] | milvus healthz |
| LLM JSON 不合法 | 节点用 fallback | `_load_json` 兜底 | 看对应节点 draft 日志 |
| LLM 编造引用 | 引用被删 | `_validate_and_fix_citations` | 合法 source_index |
| 前端 SSE 粘包处理 bug | 解析失败 | 依赖 buffer 拆包 | 看浏览器 console |
| 超 300s | 前端断流 | Nginx read timeout | 调 timeout 或减 iteration |
| 同 thread 并发双请求 | 状态互相干扰风险 | 共享 thread_id checkpoint | 请求级唯一 thread 或加锁 |

## 15. 小结：这些细节为什么重要

1. **懒加载**让容器“看起来 healthy”，但首次请求才真正连齐依赖。  
2. **前端 SSE buffer** 决定流式稳定性；**Thread ID** 决定记忆是否串话。  
3. **主图不绑 tools**，外部检索是节点硬编码调用，不是 LLM 自由 function call。  
4. **Token 隔离**保证长链路可控，记忆靠 `memory_context` 而非无限 messages。  
5. **失败默认降级**（空检索/fallback JSON/内存 checkpoint）优先保证“能出报告”，代价是质量可能下降——排障要看日志而不是只看 final。
