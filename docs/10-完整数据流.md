# 10 - 完整数据流（端到端追踪）

## 1. 概述

本章从用户输入一句话开始，逐层追踪数据如何流经系统的每一个组件，直到最终报告返回给用户。覆盖两条路径：**直接回答**（direct）和**深度研究**（multiagent）。

---

## 2. 场景一：直接回答路径

**用户输入**："你是谁？"

### 2.2 耗时分析

| 阶段 | 耗时（估算） | 说明 |
|------|-------------|------|
| 初始化（首次） | 5-10s | Agent 构建、MySQL 建表、Redis 连接 |
| 记忆注入 | 0.5-2s | Milvus 检索 + MySQL 查询 |
| intent 节点 | 1-2s | 规则引擎 + LLM 调用 |
| direct_answer 节点 | 2-5s | LLM 生成回答 |
| 记忆持久化 | 0.5-1s | Redis/MySQL 写入 |
| **总计** | **4-10s**（首次） / **3-8s**（后续） | |

### 3.2 耗时分析

| 阶段 | 耗时（估算） | 说明 |

## 4. 场景三：深度研究路径（多轮迭代）

### 4.1 迭代流程

```

                                                              │
                                                    needs_more_research=true
                                                    missing_gaps=["缺少竞争对手对比"]
                                                              │
                                                              ▼
                                                         reflect
                                                    iteration: 0→1
                                                    supplementary_queries: [
                                                      {query: "AI Agent 框架对比 LangChain CrewAI", ...},
                                                      {query: "AI Agent 竞品分析 2025", ...}
                                                    ]
                                                              │
                                                         ┌────┴────┐
                                                         ▼         ▼
第 2 轮：                                          web_search   local_rag
                                              (使用补搜计划)  (使用补搜计划)
                                                         │         │
                                                         └────┬────┘
                                                              ▼
                                                        deep_dive
                                                              │
                                                              ▼
                                                        analyze
                                                    needs_more_research=false
                                                              │
                                                              ▼
                                                         write → END
```

### 4.2 关键变化

| 对比项 | 第 1 轮 | 第 2 轮（补搜） |
|--------|---------|----------------|
| iteration | 0 | 1 |
| 搜索计划来源 | `search_plan` | `supplementary_queries` |
| source_id 前缀 | `WEB1_*`, `LOC1_*` | `WEB2_*`, `LOC2_*` |
| 证据累加 | 替换 | 追加到已有 evidence |

### 4.3 证据累加机制

```python
# web_search_node 中
existing_evidence = state.get("web_evidence", [])
return {
    "web_evidence": existing_evidence + evidence,  # 累加！
}
```

第 2 轮的证据会追加到第 1 轮的证据之后，不会覆盖。最终 `evidence_pool` 包含所有轮次的证据。

## 5. SSE 流式输出数据流

### 5.1 线程模型

```
前端 (JavaScript)
    │
    │ fetch("/api/v1/research/stream", {method: "POST", body: ...})
    │ reader = response.body.getReader()
    │
    ▼
Nginx (proxy_buffering off)
    │
    │ proxy_pass http://backend:8000
    │
    ▼
FastAPI (main thread, async event loop)
    │
    │ StreamingResponse(event_stream(), media_type="text/event-stream")
    │
    ├── event_stream() 异步生成器
    │   └── yield f"data: {json.dumps(event)}\n\n"
    │
    └── worker 线程（daemon）
        └── _run_sync_with_events(query, ..., emit=emit)
            │
            │ for update in app.stream(state, config, stream_mode="updates"):
            │     for node_name, node_output in update.items():
            │         emit({"type": "phase", "node": node_name, ...})
            │
            └── emit → queue.put(event) → event_stream() yield
```

### 5.2 事件类型详解

| 事件类型 | 触发时机 | 数据结构 |
|----------|----------|----------|
| `status` | 请求进入时（API 层直接发送） | `{"type":"status","message":"任务已接收"}` |
| `phase` | 每个节点执行完成后 | `{"type":"phase","node":"plan","message":"Planner 正在拆解问题"}` |
| `route` | 工作流执行完成时 | `{"type":"route","message":"已走多智能体研究路径"}` |
| `final` | 工作流执行完成时 | `{"type":"final","query":"...","final":"..."}` |
| `error` | 执行异常时 | `{"type":"error","message":"..."}` |

### 5.3 前端接收示例

```javascript
const response = await fetch("/api/v1/research/stream", {
  method: "POST",
  headers: {"Content-Type": "application/json"},
  body: JSON.stringify({query: "AI Agent 发展趋势"}),
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const {done, value} = await reader.read();
  if (done) break;
  
  const text = decoder.decode(value);
  const lines = text.split("\n\n").filter(line => line.startsWith("data:"));
  
  for (const line of lines) {
    const data = JSON.parse(line.replace("data: ", ""));
    
    switch (data.type) {
      case "status":
        showStatus(data.message);
        break;
      case "phase":
        showPhase(data.node, data.message);
        break;
      case "route":
        showRoute(data.message);
        break;
      case "final":
        renderMarkdown(data.final);
        break;
      case "error":
        showError(data.message);
        break;
    }
  }
}
```

## 7. 引用校验数据流

### 7.1 完整引用链路

```
证据收集阶段：
  Bocha API → raw_records → source_id = "WEB1_1-1"
  Milvus    → raw_records → source_id = "LOC1_1-1"

证据裁判阶段：
  deep_dive → evidence_pool（带 source_id）
  deep_dive → source_index（source_id + label + locator）

分析阶段：
  analyze → findings（每个 claim 绑定 source_ids）

写作阶段：
  write → Writer Agent 生成正文（使用 [WEB1_1-1] 格式引用）
  write → _validate_and_fix_citations（移除非法引用）
  write → _render_reference_list（生成 ## 参考资料）

最终输出：
  # 研究报告标题
  
  ## 核心摘要
  基于多源证据分析 [WEB1_1-1]，2025年 AI Agent 将...
  
  ## 详细分析
  ...
  
  ## 参考资料
  - [WEB1_1-1] [web]: 页面标题 | https://example.com
  - [WEB1_1-2] [web]: 页面标题 | https://example.com
  - [LOC1_1-1] [local]: 文件名 | /path/to/file.md
```

### 7.2 非法引用处理

LLM 可能会编造不存在的引用（如 `[WEB-10]`），`_validate_and_fix_citations` 会：

1. 提取正文中的所有引用 ID：`[WEB1_1-1]`, `[WEB-10]`, ...
2. 与合法引用 ID 列表比对
3. 保留合法引用，移除非法引用（替换为空字符串）
4. 返回修正后的内容和实际使用的引用列表

## 8. 错误处理数据流

### 8.1 各层错误处理策略

| 层级 | 错误类型 | 处理策略 |
|------|----------|----------|
| Bocha API | HTTP 错误/超时 | 返回空列表，跳过网页上下文注入 |
| Milvus | 连接失败 | 降级 MySQL/SQLite 检索 |
| LLM 调用 | 超时/格式异常 | 使用 fallback JSON 兜底 |
| JSON 解析 | 格式异常 | `_load_json` 返回 fallback |
| Redis | 连接失败 | 降级 InMemorySaver / 内存 |
| MySQL | 连接失败 | 降级 SQLite / 内存 |
| 记忆系统 | 初始化失败 | 返回 None，禁用记忆 |
| 整体工作流 | 任意节点异常 | SSE 发送 `{"type":"error","message":"..."}` |

### 8.2 Fallback 机制

每个节点都有 fallback，确保即使 LLM 输出异常也能继续执行：

```python
payload, content, messages = _invoke_json_agent(
    state, prompt, agent, agent_name, "plan", fallback,
)
```

Fallback 是预先定义的默认 JSON 结构，包含：
- 默认大纲
- 默认子问题（原始查询本身）
- 默认预算
- 默认证据列表（直接从原始记录构建）
